# Phase 2 – Step 1

# LLM Provider Integration (Bring-Your-Own-Key)

## Goal

Let the user answer natural-language questions about their connected
databases ("Which table stores customer information?") using a **hosted
LLM provider of their choice** — Anthropic (Claude), OpenAI (GPT), Google
(Gemini), or Groq — authenticated with their own API key.

This **revises** the original roadmap. Every prior phase-1 doc's "Next Step"
section pointed at *local* LLM integration via Ollama
(`docs/phase-1/step-9.md`, `step-10.md`), and AGENTS.md's stated stack
listed "Ollama, Any free available models." That plan is replaced here —
no local model is run by this application. The user brings their own
provider and key.

---

# Why This Step Exists

The original reasoning for Ollama was learning-oriented: no API cost, fully
local, good for understanding how inference actually works. The tradeoff
that reasoning under-weighted:

* Local models small enough to run without a GPU are meaningfully worse at
  the actual task (schema reasoning, SQL generation in later phases) than
  current hosted frontier models.
* This project's own vision statement calls it "production-inspired" —
  production systems generating SQL from natural language overwhelmingly
  use hosted frontier models, not a locally-run small model.
* Users already have accounts/keys with these providers in the common case
  worth building for, and different providers have different strengths —
  hardcoding one (or requiring local infra) is more restrictive than
  letting the user choose.

---

# Real World Example

Before (original plan):

```
User → App → Ollama (local, one fixed small model) → Answer
```

After (this step):

```
User registers a provider + API key (e.g. Anthropic, their own key)
    ↓
User asks: "Which table stores customer information?"
    ↓
App builds context from the metadata catalog (Steps 4-9)
    ↓
App calls the user's chosen provider with that context
    ↓
Answer, generated by whichever model the user picked
```

Two different users of this same codebase could each point it at a
different provider (one at Claude, one at GPT) without any code change —
only their own stored provider config differs.

# Key Concepts

## Provider Abstraction (same pattern as Phase 1 Step 11)

Exactly the lesson learned from `SourceConnector`
(`docs/phase-1/step-11.md`): don't hardcode one vendor's SDK/API shape
into every call site. Define one `LLMProvider` interface
(`app/llm/base.py`) — `generate(prompt, system=None) -> str` at minimum —
and one implementation per provider (`AnthropicProvider`, `OpenAIProvider`,
`GeminiProvider`, `GroqProvider`), selected by a factory keyed on a
`provider` string, the same way `get_connector(dialect, ...)` already
works for databases.

Note: Groq (the LPU-based inference company hosting open models like
Llama — a different company from xAI's "Grok" model, easy to mix up given
the near-identical name) has an OpenAI-SDK-compatible API (same
request/response shape, different base URL) — `GroqProvider` reuses
`OpenAIProvider`'s logic entirely with a different `base_url` and model,
not a fully separate implementation.

## Bring-Your-Own-Key

The app never ships with or defaults to any provider's key. The user
registers one (or more) via the API, the same way they register a database
connection — and it's encrypted at rest using the exact mechanism already
built for database passwords (`app/core/security.py`, Fernet). No new
encryption approach needed; reuse it.

## One Selected Provider Per Ask (Not Automatic Multi-Provider Routing)

A user may register multiple providers, but each question is answered by
one explicitly chosen (or default) provider — no automatic fallback or
cost-based routing between providers in this step. That's a possible
future step, not required to unblock asking questions at all.

# Architecture

```
POST /ai/providers  (register: provider name + API key + default model)
    │
    ▼
AIProviderConfig (Postgres) — API key encrypted via app/core/security.py
                               (same Fernet key/mechanism as connection passwords)

POST /connections/{id}/ask  {"question": "...", "provider_id": optional}
    │
    ▼
Build context from metadata catalog (tables/columns/relationships/docs —
already built in Phase 1)
    │
    ▼
get_llm_provider(provider_config) → LLMProvider
    │
    ▼
provider.generate(prompt=question + schema context)
    │
    ▼
Answer returned to user
```

# API Endpoints (planned)

## Register a Provider

```
POST /ai/providers
{
  "provider": "anthropic",       // anthropic | openai | gemini | groq
  "api_key": "sk-...",
  "default_model": "claude-..."  // optional, provider-specific
}
```
Response never echoes the key back (same pattern as `ConnectionResponse`
excluding passwords).

## List Registered Providers

```
GET /ai/providers
```
Returns provider name, default model, created_at — never the key.

## Ask a Question

```
POST /connections/{connection_id}/ask
{
  "question": "Which table stores customer information?",
  "provider_id": 1   // optional — falls back to a default if omitted
}
```

# Explicitly Out of Scope (This Step)

* No local LLM / Ollama support — this step replaces that plan entirely,
  it doesn't add to it.
* No streaming responses — a single request/response `generate()` call
  first; streaming is a follow-up if needed.
* No automatic multi-provider fallback or cost-based routing.
* No fine-tuning or prompt-caching optimization — first version prioritizes
  a correct, simple call to whichever provider is selected.
* No token/cost usage tracking yet — worth a future step once real usage
  patterns exist to design around.
* Text-to-SQL generation itself is a later phase (`Phase 5` in the original
  roadmap) — this step only wires up "ask a question, get an answer using
  schema context," not "generate and run SQL."

# Implementation Requirements

### Core

* `app/llm/base.py` — `LLMProvider` ABC
* `app/llm/anthropic_provider.py`, `openai_provider.py`, `gemini_provider.py`,
  `groq_provider.py`
* `app/llm/factory.py` — `get_llm_provider(provider, api_key, model)`,
  rejects unsupported provider names clearly (same pattern as
  `app/connectors/factory.py`'s dialect check)

### Model / Storage

* `AIProviderConfig` model + Alembic migration — `provider`, `api_key`
  (Fernet-encrypted, reusing `app/core/security.py`), `default_model`,
  timestamps
* `AIProviderRepository`

### Services & API

* `AIProviderService` — register/list/delete
* `QAService` — assembles schema context (reusing existing
  documentation/search services from Phase 1), calls the resolved provider
* `POST /ai/providers`, `GET /ai/providers`, `DELETE /ai/providers/{id}`
* `POST /connections/{connection_id}/ask`

### Dependencies

Add only the SDKs actually used: `anthropic`, `openai` (also covers Groq,
being OpenAI-compatible), `google-genai` (the current SDK — the older
`google-generativeai` package is fully deprecated). Per AGENTS.md's
existing rule, don't add any of these ahead of actually implementing that
provider's class.

# Deliverables

* A user can register any of the four supported providers with their own
  key.
* A user can ask a natural-language question about a connection's schema
  and get an answer generated by their chosen provider.
* Adding a fifth provider later is one new class + one factory
  registration — same shape as the database-connector abstraction.

# Next Step

## Phase 2 – Step 2

### Schema Context Assembly for Prompts

Now that questions can reach an LLM, the next problem is *what* context to
send it — a full schema dump doesn't scale past a handful of tables. This
step is about selecting and formatting the relevant subset of tables/
columns/relationships (likely using the search infrastructure from Phase 1
Step 7) into a prompt that fits the provider's context window and actually
helps it answer correctly.
